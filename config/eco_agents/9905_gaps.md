## GAP-F: Non-Standard ECO Naming Convention

**Status:** Show-stopper. CRITICAL_RULES Rule 10b mandates `<target_register>_reg` instance naming and `<target_register>` Q-output net naming for FM DFF auto-matching. ECO 9905 uses a completely different convention.

**Discovered in:** DEUMCIPRTL-9905 assessment (2026-05-11).

### Engineer's naming

**DFF instances:** `AMDLCECO_CECO_9905_REG_RdRdCtr_4_0__0_` (where trailing `_0_` is the bit index)  
**Q output nets:** `CECO_9905_RdRdCtr_4_0__0_` (DFF instance name without `AMDLCECO_` prefix and `REG_` stripped to a net name)  
**Intermediate wires:** `CECO_9905_1`, `CECO_9905_2`, ..., `CECO_9905_4410` (sequential integers)  
**ECO cells:** `CECO_9905_0001` through `CECO_9905_0449` (zero-padded 4-digit)  
**Clock gates:** `AMDLCECO_CECO_9905_REG_RdRdCtrCg_4` through `..._7`

### What goes wrong

**FM auto-matching (FmEqvEcoSynthesizeVsSynRtl):** Formality's `read_svf` + auto-match uses instance names to pair REF (RTL) DFFs with IMPL (gate-level) DFFs. The convention `<target_register>_reg` (instance) + `<target_register>` (Q net) is what SVF guidance is tuned to. `AMDLCECO_CECO_9905_REG_RdRdCtr_4_0__0_` as instance + `CECO_9905_RdRdCtr_4_0__0_` as Q net will NOT auto-match ‚Äî FM will report 53 unmatched compare points ‚Üí `FmEqvEcoSynthesizeVsSynRtl` FAILS.

**Our flow's naming:** `eco_applier` uses `eco<jira>_<seq>` for cells and `n_eco_<jira>_<seq>` for intermediate wires. DFF instances follow `<target_register>_reg` (Rule 10b). None of these match the engineer's CECO-style naming.

**SVF file:** `data/svf/` contains a pre-authored SVF that maps the CECO naming to RTL. Our flow would generate a new SVF using our naming convention ‚Äî the FM session that reads `data/svf/` would get naming conflicts.

### What needs to change

1. **`rtl_diff_analyzer.md`** ‚Äî add `eco_naming_convention` field to every `new_logic_dff` entry:
   - `"standard"` ‚Üí follows `<target_register>_reg` / `n_eco_<jira>_*` convention (9868 style)
   - `"ceco_style"` ‚Üí follows `AMDLCECO_CECO_<jira>_REG_*` / `CECO_<jira>_N` convention (9905 style)
   - Detect by: if RTL diff shows existing `AMDLCECO_*` or `CECO_*` prefixed instances in PostEco netlist, classify as `ceco_style`

2. **`eco_netlist_studier.md`** ‚Äî when `eco_naming_convention == "ceco_style"`:
   - DFF instance name: use the literal name from the PostEco netlist (don't generate a new name)
   - Q output net: use the literal Q net from PostEco (don't generate `<target_register>`)
   - Intermediate wires: use `CECO_<jira>_N` sequential format
   - ECO cells: use `CECO_<jira>_NNNN` zero-padded format

3. **`CRITICAL_RULES.md`** ‚Äî Rule 10b needs an EXCEPTION clause:
   - "Exception: when `eco_naming_convention == 'ceco_style'` (detected from PostEco netlist), use the literal names from the PostEco netlist verbatim. FM auto-matching will use the existing SVF in `data/svf/` which is pre-authored for the CECO naming."

4. **`eco_validate_step1.py`** ‚Äî new check `[ECO_NAMING_CONSISTENCY]`:
   - If rtl_diff has `eco_naming_convention == "ceco_style"` entries, verify NO entries also use standard naming
   - Mixed convention ‚Üí FAIL (cannot have both conventions in the same ECO)

5. **`eco_pre_fm_check.py`** ‚Äî new check `[CECO_NAMING_SVF_MATCH]`:
   - For `ceco_style` ECOs, verify `data/svf/*.svf` exists and contains `AMDLCECO_CECO_<jira>_REG_` entries
   - If SVF missing ‚Üí FAIL (FM will not auto-match any DFFs)

### Effort

Medium (3-4 hours). Mostly classification + pass-through of engineer names. The hardest part is teaching the studier NOT to generate its own names when ceco_style is detected.

### Workaround until implemented

Manual Step 3: after auto-study generates entries with standard names, hand-replace all instance/net names with the CECO-style names from the PostEco netlist. Verify with zdiff that names match exactly.

## GAP-G: Large-Scale ECO (Scale Handling)

**Status:** Show-stopper. The flow has no mechanism for handling ECOs at this scale (53 DFFs, 449 cells, 4,410 wires, 16k diff lines). Context window exhaustion and sequential processing timeouts will cause partial or empty output at every step.

**Discovered in:** DEUMCIPRTL-9905 assessment (2026-05-11).

### Scale numbers

| Item | 9868 | 9905 | Ratio |
|---|---|---|---|
| RTL diff lines | ~50 | ~800 | 16√ó |
| New DFF instances | 2 | 53 | 26√ó |
| New comb cells | 7 | 449 | 64√ó |
| New wire decls | ~10 | 4,410 | 441√ó |
| Synth diff lines | ~200 | 16,443 | 82√ó |
| PP diff lines | ~150 | 15,993 | 107√ó |

### What goes wrong per step

**Step 1 (rtl_diff_analyzer):**  
Reading and analyzing 800 lines of RTL diff with 53 new register groups pushes the agent toward context exhaustion. The agent may:
- Emit only the first few DFF groups and omit the rest
- Conflate array expansion entries (e.g., emit one entry for `RdRdCtr` when there are 4 separate entries for indices [4], [5], [6], [7])
- Time out on the gate-chain decomposition for 449 INV cells

**Step 2 (fenets_runner):**  
Cat 4 query derivation for 53 DFF entries √ó (leaves per DFF) = potentially 300-500 FM queries. FM `find_equivalent_nets` with 500 nets takes 30+ minutes per stage. Multiple stage √ó retry cycles could add up to 3+ hours just for fenets.

**Step 3 (eco_netlist_studier):**  
Processing 53 DFF entries √ó 3 stages = 159 study entries, each with gate chains. Reading the full 16k-line Synth diff to verify each entry pushes far past any context budget. Agent will:
- Produce a truncated study JSON (first N entries only)
- Fail to verify per-stage wire names for most entries

**Step 4 (eco_applier):**  
eco_passes_2_4.py processes changes sequentially. 4,410 wire declarations + 449 cell insertions + 53 DFF insertions in a single pass = ~5,000 file operations on a ~65MB gzipped file. Sequential write-read-write cycles will take hours and may corrupt the netlist if interrupted.

### What needs to change

1. **`rtl_diff_analyzer.md`** ‚Äî add `array_expansion` change type for register array expansions:
   - Trigger: RTL diff shows new always/generate blocks with array indices `[old_width:new_width-1]`
   - Emit ONE `array_expansion` entry per register array (not one entry per bit/index)
   - Include: `base_name`, `old_width`, `new_width`, `new_indices[]`, `gate_chain_template` (one chain template ‚Äî all entries share the same topology)
   - Benefit: 4 entries per array √ó 4 register arrays = 16 rtl_diff entries instead of 53

2. **`eco_netlist_studier.md`** ‚Äî add `array_expansion` Phase 0 branch:
   - Generate study entries by INSTANTIATING the `gate_chain_template` for each `new_index`
   - Do NOT manually trace each entry ‚Äî expand programmatically from the template
   - Emit a compact study format: `expansion: { base_name, indices[], chain_template }` instead of 53 individual entries

3. **`eco_passes_2_4.py`** ‚Äî add `apply_array_expansion()` batch function:
   - Accepts the `array_expansion` study entry
   - Generates all wire decls + cell instances + DFF instances in one pass
   - Writes them as a single bulk insert before `endmodule`
   - Avoids 5,000 sequential edits by collecting all insertions then writing once

4. **`eco_fenets_derive_queries.py`** ‚Äî for `array_expansion` entries:
   - Query the `gate_chain_template` leaf inputs ONCE (not once per index)
   - Clock and reset signals queried once per array (not 53 times)
   - Saves ~95% of query volume

5. **`ORCHESTRATOR.md`** ‚Äî add scale detection gate:
   - After Step 1, check if `len(rtl_diff["changes"]) > LARGE_SCALE_THRESHOLD` (e.g., 30)
   - If exceeded: switch to `large_scale_mode` which spawns Step 3 in CHUNKED mode (process N entries at a time, write partial study JSONs, merge at end)

6. **New script `eco_expand_array.py`**:
   - Reads `array_expansion` entries from rtl_diff
   - Programmatically expands them into full wire decls + cell inserts + DFF insts
   - Writes directly to PostEco netlist without going through agent reasoning
   - This is pure mechanical expansion ‚Äî no AI judgment needed

### Effort

Large (1-2 days). Core is `array_expansion` type end-to-end + `eco_expand_array.py` script. The script handles the bulk of the work mechanically, bypassing the agent for scale-sensitive operations.

### Workaround until implemented

Do NOT attempt the full flow on 9905. Either:
1. Use `eco_expand_array.py` (once implemented) standalone, bypassing the agent flow entirely for this ECO
2. Run the flow on a SUBSET (e.g., only `RdRdCtr[4]` entries) to validate the pattern, then script-expand the rest

## GAP-H: Route Stage No-Op Handling

**Status:** Bug. The flow always processes all 3 stages (Synthesize, PrePlace, Route). When Route PostEco = PreEco identically, several validators and checks incorrectly flag it as an error or try to apply changes that don't exist.

**Discovered in:** DEUMCIPRTL-9905 assessment (2026-05-11). PostEco `Route.v.gz` (72,904,922 bytes, Mar 25 04:52) is bit-for-bit identical to PreEco `Route.v.gz`. The engineer simply did not apply the ECO to the Route stage.

### What goes wrong

**`eco_validate_step3.py` Check 1:** "All 3 stages present and non-empty." If studier correctly produces empty Route entries (no changes), Check 1 fails with "CRITICAL: Route entries empty."

**`eco_validate_step4.py` Check 3:** "PostEco must differ from PreEco for stages with insertions." This check PASSES correctly for Route (no insertions ‚Üí no MD5 check). But Check 5 ("All confirmed study entries are accounted for") will fail if Route study entries are non-empty but applier skips them.

**`eco_pre_fm_check.py` `[ALL_STAGES_CHANGED]` check** (if present): May flag Route as unchanged ‚Üí block FM submission.

**FM submission itself:** `FmEqvEcoRouteVsEcoPrePlace` compares PostEco Route vs PostEco PrePlace. If Route = PreEco (unchanged), this FM check compares PreEco Route vs PostEco PrePlace ‚Äî these are genuinely different (PreEco Route ‚âÝ PostEco PrePlace) ‚Üí FM FAILS with real differences, not ECO-related ones.

### What needs to change

1. **`rtl_diff_analyzer.md`** ‚Äî add `route_stage_noop: true` field to the rtl_diff JSON output:
   - Set when: `zdiff data/PreEco/Route.v.gz data/PostEco/Route.v.gz` produces no output (identical)
   - OR: MD5 of PreEco Route = MD5 of PostEco Route

2. **`eco_netlist_studier.md`** ‚Äî when `route_stage_noop: true`:
   - Emit an EMPTY Route section `"Route": []` with a `"route_noop": true` note
   - Do NOT try to match any study entries for Route
   - Explicitly: "Route stage is no-op ‚Äî engineer did not apply ECO to Route. Emit empty Route section."

3. **`eco_validate_step3.py`** ‚Äî Check 1 must be gated:
   ```python
   route_noop = rtl_diff.get('route_stage_noop', False)
   for stage in ['Synthesize', 'PrePlace', 'Route']:
       if stage == 'Route' and route_noop:
           continue  # empty is correct for no-op Route
       entries = study.get(stage, [])
       if not entries:
           issues.append(f"CRITICAL: {stage} entries empty ...")
   ```

4. **`eco_validate_step4.py`** ‚Äî Check 3 (MD5 unchanged = error) must be gated:
   ```python
   if stage == 'Route' and applied.get('route_noop', False):
       continue  # Route unchanged is expected
   ```

5. **`eco_pre_fm_check.py`** ‚Äî new check `[ROUTE_NOOP_CONSISTENT]`:
   - If `route_stage_noop: true` in study: verify PostEco Route MD5 = PreEco Route MD5
   - If they differ: FAIL (engineer said no-op but Route was actually modified ‚Äî inconsistency)
   - If they match: PASS (no-op is legitimate)

6. **`ORCHESTRATOR.md`** ‚Äî Step 4 FM submission gate:
   - Check `route_stage_noop` flag before submitting `FmEqvEcoRouteVsEcoPrePlace`
   - If true: SKIP Route FM submission entirely ‚Üí mark as PASS_SKIPPED
   - Document: "Route FM skipped ‚Äî engineer did not apply ECO to Route stage (PostEco Route = PreEco Route)"

### Effort

Small (1-2 hours). Mostly a flag that gates existing checks. Clean and low-risk.

### Workaround until implemented

Manual: Before running FM, copy PostEco PrePlace ‚Üí PostEco Route so that `FmEqvEcoRouteVsEcoPrePlace` compares identical files ‚Üí trivially passes. This is technically wrong (introduces a fake Route) but avoids FM failure while the real fix is implemented.

## GAP-I: Clock Gate Cell Handling

**Status:** Not supported. The flow's `apply_new_cell()` handles standard combinational cells (INV, AND, OR, NAND, etc.) and DFFs. Clock gate cells (`CKOR2LNQAN2D1AMD`) have a different pin topology and semantics that require separate handling.

**Discovered in:** DEUMCIPRTL-9905 assessment (2026-05-11). 9 new clock gate instances with:
```verilog
CKOR2LNQAN2D1AMDBWP136P5M156H3P48CPDLVTLL AMDLCECO_CECO_9905_REG_RdRdCtrCg_4 (
  .CK(UCLK01G0),   // source clock
  .TE(1'b0),        // test enable (scan)
  .Q(CECO_9905_RdRdCtrCg_4_Q)  // gated clock output
);
```

### What goes wrong

**`rtl_diff_analyzer`:** Clock gate RTL is typically `if (enable) CK` or a `GCLK` instantiation. The analyzer may misclassify it as a `new_logic_dff` (it has a `.Q` output) or as a `new_logic_gate_only` (combinational) ‚Äî neither is correct.

**`eco_netlist_studier` Phase 0:** Clock gates have `.CK` (not `.D`), `.TE` (not `.SE`), `.Q` (gated clock output not a data flip-flop Q). Mode S triage doesn't apply. The D-input chain concept is wrong for a clock gate.

**`eco_pre_fm_check.py`:** Clock gate Q outputs drive DFF `.CP` pins ‚Äî not data paths. The check for `new_logic` chain correctness won't understand clock-gate-driven clock nets.

### What needs to change

1. **`rtl_diff_analyzer.md`** ‚Äî add `new_clkgate` change type:
   - Trigger: RTL diff shows a new `GCLK` / `CKO*` / `ICG*` cell instantiation
   - Required fields: `clkgate_inst`, `source_clock`, `enable_signal`, `gated_clock_output`, `te_value` (scan test enable)
   - No `d_input_gate_chain`, no `reset_signal`, no `dff_clock`

2. **`eco_netlist_studier.md`** ‚Äî add `new_clkgate` Phase 0 branch:
   - Resolve `source_clock` per stage via fenets rename map (CTS may rename it)
   - Resolve `enable_signal` per stage
   - Emit study entry with `strategy: "clkgate_insertion"` ‚Äî no Mode S triage

3. **`eco_passes_2_4.py`** ‚Äî new `apply_new_clkgate()` function:
   - Insert clock gate cell before `endmodule` in target module
   - Wire: `.CK(<source_clock_per_stage>), .TE(<te_value>), .Q(<gated_clock_output>)`
   - Add wire declaration for `gated_clock_output` if it doesn't exist

4. **`eco_validate_step1.py`** ‚Äî new check `[CLKGATE_FIELDS]`:
   - For `new_clkgate` entries: verify `source_clock`, `gated_clock_output` are present
   - FAIL if `d_input_gate_chain` is also present (incompatible with clkgate)

### Effort

Small-medium (2 hours). Clock gate structure is simple and regular. Main complexity is ensuring the gated clock Q output is recognized as a clock net (not a data net) in downstream checks.

## GAP-J: Self-Referential Scan (SI=Q Pattern)

**Status:** Not in Mode S playbook. Current Mode S strategies are: `constant_zero` (SI=1'b0, SE=1'b0), `neighbor_dff` (SI/SE from a nearby DFF), `bridge_port` (SI/SE from cross-module bridge). `SI=Q_of_self, SE=1'b0` (scan-isolated DFF) is a fourth pattern not documented or handled.

**Discovered in:** DEUMCIPRTL-9905 assessment (2026-05-11). All 53 new DFF instances use:
```verilog
SDFQD1AMD RdRdCtr_reg_N (
  .D  (<computed_value>),
  .SI (<Q_output_of_self>),  // Q of the SAME DFF ‚Äî self-referential scan loop
  .SE (1'b0),                // scan enable permanently disabled
  .CP (<gated_clock>),
  .Q  (<Q_output_of_self>)
);
```

### What goes wrong

**`eco_netlist_studier` Mode S triage:**  
Synth stage: SE=1'b0, SI=Q_of_self. The triage sees `constant_zero` for SE but `Q_of_self` for SI (not a constant, not a neighbor). Current strategy map has no rule for `SI=own_Q_output` ‚Üí studier may default to `neighbor_dff`, look for a nearby DFF's Q, pick the wrong wire ‚Üí FM PP/Route failure on scan cone divergence.

**`eco_passes_2_4.py`:**  
When applying Mode S, the code substitutes the resolved SI/SE values. For `self_loop` mode, SI = the DFF's own Q output net (which is also being declared in this same insertion). If the Q net name is derived before it's declared, the substitution may produce a forward reference or an incorrect wire name.

### What needs to change

1. **`eco_netlist_studier.md`** ‚Äî add `self_loop` to Mode S strategy enum:
   - Strategy name: `"mode_s_self_loop"`
   - Detection: `.SI` wire in PostEco netlist is the same net as `.Q` of that DFF instance
   - Emit: `si_source: "SELF_Q"`, `se_value: "1'b0"` (constant)
   - All 3 stages use the same pattern (no PP bridge, no Route bridge)

2. **`eco_passes_2_4.py`** ‚Äî `apply_new_cell()` for DFFs ‚Äî handle `mode_s_self_loop`:
   ```python
   if entry.get('mode_s_strategy') == 'mode_s_self_loop':
       si_value = entry['q_output_net']   # SI = own Q
       se_value = "1'b0"
   ```
   This is deterministic ‚Äî no fenets query needed for SI (it's the same as Q).

3. **`eco_validate_step3.py`** ‚Äî new check `[MODE_S_SELF_LOOP]`:
   - For any study entry with `mode_s_strategy == 'mode_s_self_loop'`, verify `si_source == "SELF_Q"`
   - FAIL if SI references any other DFF's Q (self-loop must reference own Q only)

4. **`eco_pre_fm_check.py`** ‚Äî new check `[SELF_LOOP_SCAN_DISABLED]`:
   - For `mode_s_self_loop` DFFs, verify `.SE(1'b0)` in PostEco (not a wire)
   - Verify `.SI(<Q_net>)` and `.Q(<Q_net>)` use the SAME net name
   - FM handles self-loop SI gracefully (it's a known scan-disable pattern) ‚Äî no special FM action needed

### Effort

Small (1-2 hours). Purely additive ‚Äî new strategy enum value + deterministic SI assignment. No FM query needed for SI (it's just `own_Q`).
