## GAP-A: No-DFF ECO Type (`new_logic_gate_only`)

**Status:** Not supported. Flow assumes every ECO has at least one new DFF as an anchor for Step 3 (studier) Phase 0.

**Discovered in:** DEUMCIPRTL-9899 assessment (2026-05-11). ECO 9899 inserts 9 combinational cells in umccmdarb, 5 cells in each DCQARB, zero scan flip-flops anywhere.

### What goes wrong

`eco_netlist_studier` Phase 0 is the DFF-insertion phase — it is the entry point for all `new_logic` classification. Phase 0 expects:
1. A target register name (the new DFF's instance name) to anchor the chain
2. A `dff_clock` field to resolve per-stage CTS renames
3. A `reset_signal` field for scan chain Mode S derivation
4. A `d_input_gate_chain[]` rooted at the DFF's `.D` pin

For 9899, none of these exist. The ECO is a floating combinational cone that replaces the output driver of an existing net (`ctmn_2084955`) with a chain of 9 new ECO cells. Phase 0 has no register to anchor on → studier may skip the entire `new_logic` entry or emit an incomplete study JSON, causing applier to apply nothing.

Additionally, `eco_fenets_derive_queries.py` Cat 2/3/4 (clock, reset, chain leaves) are all gated on `new_logic` entries having `dff_clock` or `reset_signal` fields. If those are absent, the query list shrinks to Cat 1 only (wire_swap/and_term) → fenets underqueries → studier gets incomplete rename map.

### What needs to change

1. **`rtl_diff_analyzer.md`** — add `new_logic_gate_only` change type (distinct from `new_logic` and `new_logic_dff`):
   - No `dff_clock`, no `reset_signal`, no Mode S section
   - Has `output_net` (the net driven by the new cell chain's final output)
   - Has `replaces_net` (the old net whose drivers the new chain replaces)
   - Has `gate_chain[]` from input leaves to output net
   - Classification trigger: RTL diff shows new combinational assignments or new cell instantiations with NO new register

2. **`eco_netlist_studier.md`** Phase 0 — add a `new_logic_gate_only` branch:
   - Skip DFF anchor search
   - Find old net (`replaces_net`) in PostEco stage → verify it exists (single-occurrence rule still applies)
   - Locate gate-level name of `replaces_net` from fenets rename map
   - Emit study JSON entry with `strategy: "cone_replacement"`, `output_net`, `gate_chain[]`

3. **`eco_fenets_derive_queries.py`** — add Cat 9: `new_logic_gate_only` entries:
   - Query `replaces_net` (to get its gate-level equivalent — the cell it drives will be the insertion point)
   - Query every leaf input in `gate_chain[].inputs`
   - Skip Cat 2/3 (no clock/reset)

4. **`eco_passes_2_4.py`** — new `apply_cone_replacement` action:
   - Insert new gate cells before the `endmodule` line of the target module
   - Rewire the consuming cell's input (the cell that used to receive `replaces_net`) to receive the new chain's output wire
   - Do NOT delete the old net — only replace who drives the consuming cell's input pin

### Effort

Medium (half day). Mostly additive — new branch in Phase 0, new change type in Step 1, new Cat 9 in fenets derive script.

### Workaround until implemented

Manual Step 3: after auto-study runs, hand-author a `cone_replacement` entry in `<TAG>_eco_preeco_study.json` for the ctmn_2084955 chain. Applier can use the existing `new_logic` path if the entry shape is close enough.

## GAP-B: Cross-Channel Wiring (Asymmetric Instance Hookup)

**Status:** Not supported. Flow assumes each new port added to a module is driven by a wire from its own scope or direct parent. Cross-channel wiring (DCQARB0 ← signal from DCQARB1's cone, and vice versa) is a new pattern.

**Discovered in:** DEUMCIPRTL-9899 assessment (2026-05-11). Engineer hookup at umcarb (top) level:
```verilog
// Cross-channel: DCQARB0 gets Cmd1, DCQARB1 gets Cmd0
ddrss_umccmd_t_umcdcqarb_0_0 DCQARB0 ( ..., .SplitActInProgOthDcq(SplitActInProgCmd1), ... )
ddrss_umccmd_t_umcdcqarb_1_0 DCQARB1 ( ..., .SplitActInProgOthDcq(SplitActInProgCmd0), ... )
```

### What goes wrong

`eco_fenets_runner` derives queries by walking `changes[]`. For DCQARB0's new `SplitActInProgOthDcq` port, Cat 1 queries the net `SplitActInProgOthDcq` in DCQARB0's hierarchy scope. But `SplitActInProgOthDcq` in DCQARB0 is driven by `SplitActInProgCmd1` — a signal that lives in DCQARB1's cone, not DCQARB0's. FM `find_equivalent_nets` at the DCQARB0 scope may return FM-036 or find no equivalent (the signal isn't declared inside DCQARB0, it's an input port). This triggers unnecessary retry loops and may resolve to the wrong net.

At the applier level, `eco_passes_2_4.py` `apply_port_connection` handles new instance hookup but only for a single hookup value per port. Cross-channel wiring requires DIFFERENT hookup values for the same port name on two different instances of similar modules — current applier may apply the same hookup to both, causing one to be wired incorrectly.

### What needs to change

1. **`rtl_diff_analyzer.md`** — add `port_connection_cross_channel` sub-type for `port_connection` changes where:
   - Two instances of similar modules get the same new port
   - But the driving signal on each instance is different (cross-connected)
   - Emit `cross_channel: true` + `per_instance_hookup: {inst0: wire_A, inst1: wire_B}`

2. **`eco_fenets_derive_queries.py`** — for `port_connection_cross_channel` entries:
   - Query the hookup wires at the PARENT scope (not the child module scope), since the wires live at umcarb level
   - One query per instance hookup wire, scoped to the parent

3. **`eco_passes_2_4.py`** — extend `apply_port_connection` to accept `per_instance_hookup` dict:
   - When applying, match the specific instance name and use its specific hookup wire
   - Do NOT broadcast the same hookup to all instances of that module type

### Effort

Small-medium (2-3 hours). Mostly a logic extension in apply_port_connection + new query category in fenets derive script.

### Workaround until implemented

Manual Step 4: after auto-apply, hand-correct the instance hookup for one of the two DCQARB instances. Verify with zdiff that DCQARB0 has `SplitActInProgCmd1` and DCQARB1 has `SplitActInProgCmd0`.

## GAP-C: Multi-Module Symmetric + Asymmetric Stage Changes

**Status:** Partial support. Flow handles multiple modules being changed in one ECO but does not explicitly track asymmetric differences between similar instances (same module type, different instance names) across stages.

**Discovered in:** DEUMCIPRTL-9899 assessment (2026-05-11). Both DCQARB0 and DCQARB1 get the same base changes (PhArbFineGater double-invert + QualPmArbWinVld gating). BUT in PrePlace and Route, DCQARB0 gets an extra OR2 cell (`ECO_9899_cell60`) that DCQARB1 does not get.

### What goes wrong

The studier and applier work at module-type level, not instance-name level. When a change is classified for `umcdcqarb`, both `umcdcqarb_0` and `umcdcqarb_1` instances get the same treatment. The extra OR2 in DCQARB0 (PP/Route only) is stage-AND-instance specific — a combination the current framework can't express cleanly.

Specific risk: applier applies OR2 insertion to BOTH dcqarb_0 and dcqarb_1 → dcqarb_1 gets a spurious OR2 → FM fails on dcqarb_1 compare points.

### What needs to change

1. **`rtl_diff_analyzer.md`** — add `instance_scope` field to all changes (distinct from `module_scope`):
   - `instance_scope`: the specific instance path where the change applies (e.g., `ARB/DCQARB0`)
   - `applies_to_all_instances_of_module: false` flag when a change is instance-specific

2. **`eco_netlist_studier.md`** — when emitting study JSON entries for changes with `instance_scope`, tag with the specific instance. Do NOT broadcast to all instances of the same module.

3. **`eco_passes_2_4.py`** — `apply_new_cell` must honor `instance_scope`:
   - Locate the specific module declaration for that instance (by matching instance name in parent → then editing that module's body)
   - When two instances of the same module type differ (dcqarb_0 ≠ dcqarb_1), the applier must split the module body edits per instance scope

4. **`eco_validate_step3.py`** — new check: for any two instances of the same module that have DIFFERENT study entries, flag as `asymmetric_instance_changes: true` and require explicit per-instance override entries.

### Effort

Medium (3-4 hours). Core change is instance-scope tracking throughout the study→apply pipeline. Biggest risk is module body splitting when two instances of the same module type need different edits.

### Workaround until implemented

Manual Step 4: after auto-apply, inspect both dcqarb_0 and dcqarb_1 bodies — remove the OR2 from dcqarb_1 if it was erroneously applied. Verify with zdiff.

## GAP-D: Large Combinational Cone Replacement (ctmn_* Pattern)

**Status:** Not supported. Flow handles gate insertions rooted at a DFF's D-input. Replacing an existing net's entire driver cone (multiple cells feeding into `ctmn_2084955`) is a different structural operation.

**Discovered in:** DEUMCIPRTL-9899 assessment (2026-05-11). Engineer replaces the combinational logic generating `ctmn_2084955` in `umccmdarb` with 9 new ECO cells. The old cells remain (they drive other things) but the wire that previously drove `ctmn_2084955` is now driven by `ECO_9899_cell_last_output` instead.

### What goes wrong

In 9868 the pattern was: insert new cells + connect output to new DFF's `.D` pin. Removing an existing net's driver and replacing with a new chain is structurally different:

- The "old_net" to replace (`ctmn_2084955`) may be driven by MULTIPLE old cells (fan-in cone) — not just one wire
- The consumer of `ctmn_2084955` is NOT a new cell — it's an EXISTING cell that must have one of its inputs rewired
- The applier's `apply_new_cell` + `apply_wire_swap` sequence was designed for DFF insertion, not mid-cone driver replacement

Specifically: the applier needs to find the EXISTING cell that consumes `ctmn_2084955` and rewrite THAT cell's pin from `ctmn_2084955` to the new ECO chain's output. The `apply_wire_swap` action does handle wire substitution but it was designed for simple net renames — a cone root replacement at an existing cell's input is structurally the same as a wire_swap, but the single-occurrence rule may block it if `ctmn_2084955` fans out to multiple consumers.

### What needs to change

1. **`rtl_diff_analyzer.md`** — when classifying a `new_logic_gate_only` change, add `consumer_cell_pin_rewire` sub-entry:
   - `consumer_cell_inst`: the existing cell that consumes the old net
   - `consumer_cell_pin`: which pin (A1, A2, etc.) currently receives `old_net`
   - `new_driver_net`: the output of the new ECO chain that replaces `old_net` on that pin
   - This makes the cone replacement self-contained — studier knows EXACTLY where to rewire

2. **`eco_passes_2_4.py`** — new `apply_consumer_rewire` action:
   - Find `consumer_cell_inst` in the current module body
   - Rewrite `.pin(old_net)` → `.pin(new_driver_net)` on that specific instance
   - Do NOT rename `old_net` globally (it may still drive other cells)

3. **`eco_pre_fm_check.py`** — new check `[CONE_REPLACEMENT_COMPLETENESS]`:
   - Verify `old_net` no longer appears as input to `consumer_cell_inst`
   - Verify `new_driver_net` appears as input to `consumer_cell_inst`
   - Verify all new ECO cells in `gate_chain[]` are present in the netlist

### Effort

Small-medium (2-3 hours). `apply_consumer_rewire` is a targeted pin-level edit — straightforward. Main complexity is step 1 (rtl_diff classifying the cone correctly).

### Workaround until implemented

Manual Step 4: after auto-apply inserts the new ECO cells, hand-locate the consumer cell in the PostEco netlist and rewrite its input pin. Verify with zdiff that `ctmn_2084955` is no longer an input to that cell.

## GAP-E: Direct Functional Port Naming (No ECO_NNNN_ Prefix)

**Status:** Assumption mismatch. Several agents and validators assume new ECO ports follow the `ECO_<jira>_*` naming convention (e.g., `ECO_9868_SI_in`, `ECO_905_SE_out`). ECO 9899 uses direct functional port names with no ECO prefix.

**Discovered in:** DEUMCIPRTL-9899 assessment (2026-05-11). New ports are:
- `SplitActInProgOthDcq` (input to DCQARB0/1) — no ECO prefix
- `PhArbFineGater` (output from DCQARB0/1) — no ECO prefix
- `DcqArb0_PhArbFineGater`, `DcqArb1_PhArbFineGater` (wires at umcarb level) — no ECO prefix

### What goes wrong

`eco_pre_fm_check.py` checks `[BRIDGE_PORT_USAGE]`, `[NO_DEAD_BRIDGE_PLUMBING]`, `[MODE_S_STAGE_STRATEGY]` all use `ECO_<jira>_*` pattern matching to identify ECO-added ports. For 9899's functional-name ports, these checks will find zero ECO ports → either silently pass (wrong) or wrongly report no ECO port changes → validator lets malformed netlist through.

Similarly, `eco_netlist_studier` port-promotion logic uses `ECO_<jira>_` as a default name prefix for newly created ports. If left as default, it would emit `ECO_9899_SplitActInProgOthDcq` instead of the correct `SplitActInProgOthDcq` → FM fails because the port name in the netlist doesn't match the RTL.

### What needs to change

1. **`rtl_diff_analyzer.md`** — emit explicit `port_name` field (not a generated name) for every `port_promotion` and `new_port` change:
   - Extract the literal port name from the RTL diff
   - Do NOT apply `ECO_<jira>_` prefix to the emitted name
   - Add `eco_naming_style` field: `"prefixed"` (ECO_NNNN_ style like 9868) or `"functional"` (direct signal name like 9899)

2. **`eco_netlist_studier.md`** — when emitting port decls, use the literal `port_name` from the rtl_diff entry:
   - Do NOT auto-prefix with `ECO_<jira>_` unless `eco_naming_style == "prefixed"`
   - Preserve the engineer's chosen name exactly

3. **`eco_pre_fm_check.py`** — extend ECO port detection to also match ports from `port_promotion` entries in the study JSON:
   - Load the study JSON
   - Build a set of all new port names (from `port_decl` entries)
   - Use that set for bridge/port checks, NOT just `ECO_<jira>_*` grep patterns

4. **`eco_validate_step3.py`** — new check: for each `port_promotion` entry in the study, verify the literal port name (not prefixed) appears in the PostEco netlist's module port list.

### Effort

Small (1-2 hours). Mostly removing the auto-prefix assumption and using literal port names throughout.

### Workaround until implemented

After auto-study generates port entries, manually correct any `ECO_9899_*` port names to use the correct functional names before running the applier.
