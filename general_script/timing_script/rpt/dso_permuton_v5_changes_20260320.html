Context Why These Changes Were Made

The V5 permuton update has two goals: (1) remove permutons that wasted DSO search budget with no proven winning value across 80+ completed lineages in the 28Jan40p and 4Mac40p runs, and (2) add permutons that directly translate the new techniques introduced in r2r_optimization.tcl v5 into explorable dimensions — letting DSO find the optimal settings for what the non-DSO tune hardcodes.

With v5 TCL applied as the DSO baseline (including DCQARB boundary opt, clock gate path groups, DCQARB max_fanout=30, and all R2R path group weight improvements), the permutons now act as parameter tuners on top of a stronger starting point rather than introducing these techniques from scratch. The target is to beat the non-DSO Mar16 result: totalCoreWNS −93.9 ps and totalCoreTNS −68,447 ps.

Non-DSO Mar16 (v3 tune): WNS = -93.889 ps | totalCoreTNS = -68,447 ps ← target to beat DSO 4Mac best (75ec3049): WNS = -85.18 ps | TNS = -137,538 ps ← WNS better, TNS much worse DSO 4Mac baseline (Feb23): WNS = -120.65 ps | TNS = -116,554 ps
Removed 3 Permutons Removed — No Winning Value
umccmd_critical_path_groups
range: {standard aggressive}

This permuton created supplemental path groups for critical R2R startpoints. Its range had two values: standard and aggressive.

Why removed:

1. aggressive always causes a catastrophic timing regression. Lineage 40b72239 (4Mac40p) with crit_groups=aggressive achieved WNS −181.21 ps — 60 ps worse than the baseline of −120.65 ps. This is the worst-performing lineage in the entire 4Mac run. The aggressive mode creates too many overlapping high-weight path groups, starving other violating groups of optimizer attention.

2. standard always wins within the range, providing zero exploration value. When the only non-crashing option always wins, the permuton contributes nothing to DSO search — it simply burns one search dimension confirming standard each time.

3. Function preserved by v5 TCL. The v5 r2r_optimization.tcl already applies comprehensive balanced path groups for all major violating groups (ARB weight=8, DCQARB weight=7, clock gating weight=6, ADDR/FEI/CMD weight=5–6). Standard-mode path group coverage is already the baseline.

umccmd_arb_r2r_opt
range: {none restructure retime}

This permuton created ARB-targeted path groups (weight 1.2–1.3) and enabled logic restructuring for ARB R2R paths to address the top R2R WNS violator.

Why removed:

1. Path group weight (1.3) is an order of magnitude weaker than v5 TCL groups (8). With the v5 TCL applied as the DSO baseline, the ARB module is already targeted at weight=8 (umc_ARB_internal_r2r) and weight=7 (umc_ARB_r2r_from). The permuton's weight-1.3 supplemental group adds negligible pressure and creates ambiguity for the optimizer about which path group to prioritize.

2. DCQARB filter bug was contaminating results. The filter full_name =~ *ARB* matched DCQARB registers (since DCQARB contains the substring "ARB"), silently mixing 49.8% of all design registers into what was intended as an ARB-only path group. Although the bug has been fixed, the permuton's results across all prior runs were unreliable for this reason.

3. Appeared in only one winning combination across all runs. The restructure value appeared in 4Mac40p best lineage (75ec3049) but was absent from all other top performers. The retime value never appeared in any winner. Insufficient evidence of consistent benefit to justify a search dimension.

umccmd_r2r_tns_weight
range: {balanced high}

This permuton tried to balance R2R WNS vs TNS by applying different path group weight schemes. Introduced in V3 to address the TNS regression seen in the 4Mac40p run.

Why removed:

1. The winning value low was removed from the range in V4 and never restored. The 4Mac40p winning lineage (75ec3049, WNS −85.18 ps) used r2r_tns_weight=low. V4 removed low due to TNS regression concerns, leaving only {balanced high} — but high has never appeared in any winning lineage across either DSO run. The current range explores values with no winning history.

2. balanced appeared in one winner but contributed nothing unique. The 4Mac Mar11 leader (b208c13c) used r2r_tns_weight=balanced, but this lineage was superseded by later lineages with better results that did not use this permuton. The balanced path group scheme it creates is already covered by v5 TCL's comprehensive multi-group setup.

3. TNS vs WNS balance is now handled by ADES metric weights. The ADES metric already penalizes TNS regression (R2R_TNS weight=3, R2R_WNS weight=2). DSO learns from ADES feedback. An explicit permuton attempting to re-balance what ADES already handles creates conflicting optimization signals.

Added 3 Permutons Added — Translate v5 TCL into Explorable Dimensions

These permutons correspond directly to the three main new techniques introduced in r2r_optimization.tcl v5. Since v5 TCL applies them with fixed values, DSO can now determine whether those fixed values are optimal or whether a different level performs better on each lineage.

umccmd_dcqarb_boundary_opt
range: {none arb_only both}

v5 TCL origin: Added set_boundary_optimization true for both ARB hierarchical cells and DCQARB hierarchical cells. This allows the optimizer to fix cross-hierarchy R2R paths without requiring logic restructuring or dont_touch=false removal (which caused instability in v2).

Why added:

1. DCQARB is the 2nd largest totalCoreTNS contributor with NVP=2,339, TNS=−39,091 ps and WNS=−62 ps. Boundary optimization allows the optimizer to treat DCQARB as a transparent boundary during timing fixing — critical for cross-hierarchy paths that previously could not be optimized.

2. v5 TCL hardcodes both (ARB + DCQARB). This permuton lets DSO verify that both is indeed optimal, or discover whether arb_only is sufficient without the additional DCQARB boundary overhead. The none option provides a control baseline to measure the gain from v5 TCL's approach.

3. Distinct from umccmd_dcq_arbiter_pipeline=restructure, which also applies boundary opt but additionally removes dont_touch and restructures DCQARB logic (aggressive, known instability risk). This permuton is the safe-only boundary opt path.

v5 TCL target (hardcoded): set_boundary_optimization $dcqarb_hier_cells true Permuton exploration: none → no boundary opt (control baseline) arb_only → ARB only (v3 behavior, confirmed safe) both → ARB + DCQARB (v5 TCL value, safe, no dont_touch removal)
umccmd_clkgate_opt
range: {none low high}

v5 TCL origin: Added group_path -name umc_clkgate_r2r -weight 6 -critical_range 400 -to $clkgate_cells targeting the clock_gating_default path group.

Why added:

1. Clock gating paths were never targeted in any DSO lineage across both prior runs. The clock_gating_default group has NVP=545, TNS=−10,175 ps, and WNS=−59 ps — the 3rd largest contributor to totalCoreTNS. Despite this, zero DSO lineages in 28Jan40p or 4Mac40p explicitly targeted this group. This represents untapped TNS reduction potential.

2. The optimal weight is unknown. v5 TCL hardcodes weight=6. Too high a weight pulls optimizer attention away from the dominant ARB (weight=8) and DCQARB (weight=7) groups. Too low adds no pressure. DSO can find the balance: low (weight=4, conservative) vs high (weight=6, v5 TCL value) vs none (let tool handle via default clock_gating_default group).

clock_gating_default (Mar16 run): NVP=545 TNS=-10,175 ps WNS=-59 ps Never targeted in 28Jan40p (43 lineages) or 4Mac40p (41 lineages) = 84 lineages of missed opportunity Permuton exploration: none → tool default clock_gating_default handling low → weight=4, critical_range=300 (conservative) high → weight=6, critical_range=400 (v5 TCL value)
umccmd_dcqarb_max_fanout
range: {none 20 30 40}

v5 TCL origin: Added set_max_fanout 30 $dcqarb_cells to reduce delays on high-fanout control nets within DCQARB.

Why added:

1. DCQARB contains 49.8% of all design registers. High-fanout control nets spanning this large block contribute directly to R2R path delays. Forcing the tool to buffer nets beyond a fanout limit is a direct lever on R2R timing.

2. The optimal fanout limit is design-state dependent and unknown. v5 TCL hardcodes 30, but this is an estimate. A tighter limit (20) forces more aggressive buffering and may improve WNS at the cost of area and potential congestion. A looser limit (40) reduces buffering overhead and may improve TNS by freeing routing resources for critical paths. DSO can determine which value produces the best ADES score across lineages.

3. Applies at both compile and place checkpoints since fanout-driven buffering affects both logic optimization (compile) and physical placement (place), giving DSO two opportunities to tune the effect.

v5 TCL target (hardcoded): set_max_fanout 30 $dcqarb_cells Permuton exploration: none → tool default (no constraint, typically 100+) 20 → aggressive buffering (fewer slow high-fanout nets, more area) 30 → v5 TCL value (moderate) 40 → relaxed (less buffering overhead, better congestion)
