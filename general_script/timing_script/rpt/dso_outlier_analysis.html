CRITICAL: dcq_arb='balance' and 'both' Break Timing Completely

100% of outliers (18 lineages with WNS > -3000ps) use dcq_arb=balance or dcq_arb=both
0% of successful lineages use these values.

Action Required: Remove 'balance' and 'both' from dcq_arb permuton search space.

Technical Root Cause: Why 'balance' and 'both' Break Timing

Analysis of umccmd_dcq_arbiter_procs.tcl reveals the fundamental issue:

if {$do_balance} {
    # Create path groups for balanced optimization with weight
    group_path -name dcqarb_group -through $dcqarb_cells -weight 2.0
    # ^^^^^^^ THIS IS THE PROBLEM ^^^^^^^
}
Issue Explanation
-through Captures ALL paths that pass through DCQ arbiter cells. DCQARB is a central arbiter with thousands of paths flowing through it.
Path Explosion Creates a massive path group with potentially millions of timing paths, overwhelming the optimizer.
weight 2.0 Forces the tool to prioritize ALL these paths with 2x weight, causing resource starvation for other timing paths.
Result Optimizer cannot converge → WNS collapses to -3000+ ps
Mode TCL Command Why it Works/Fails
restructure set_boundary_optimization Allows restructuring without path weighting - Safe
none No changes Baseline behavior - Safe
balance group_path -through ... -weight 2.0 Path explosion via -through - BROKEN
both restructure + balance Includes balance problem - BROKEN

Proposed Fix

Change -through to -to in the balance mode:

# BEFORE (broken):
group_path -name dcqarb_group -through $dcqarb_cells -weight 2.0

# AFTER (fixed):
group_path -name dcqarb_group -to $dcqarb_cells -weight 2.0

Alternative: Remove 'balance' and 'both' options entirely from the permuton search space.
